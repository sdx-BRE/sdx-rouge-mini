# Project Rules for AI Assistants

## Git Commit Conventions
All commit messages must follow this strict format:
`[<type>] - <description>`

### Rules:
- The hyphen after the closing bracket is **MANDATORY**.
- The description must start with a lowercase letter (unless it's a class/constant name).
- Allowed types are:
  - `refactoring`: For code restructurings, architecture changes, or pattern updates.
  - `fix`: For bug fixes or error corrections.
  - `task`: For new features, implementations, or logic additions.
  - `chore`: For maintenance, cleanup, type-safety updates, or meta-tasks.

## Code Standards
- **Strict Typing**: Always use static typing. Use `:=` for inference or `: <type>` for explicit declarations.
- **Clean Code**: 
  - Avoid redundant comments that explain the obvious.
  - Do not keep empty method overrides (like `pass`) unless necessary for interface compliance.
- **Naming**: Use camel_case for variables and methods, PascalCase for classes.

## Workflow Rules
- **Verification**: NEVER propose or execute a git commit command unless the user has explicitly verified the changes and given permission to commit in that specific moment.
